# ZeeSpec — хэрэглээний товч заавар (3 хувилбар)

> **Хувийн лавлах.** Гурван орох цэгийн товч жор: шинэ төсөл (идэйнээс) · хуучин код ·
> refactor. Бүрэн дэлгэрэнгүй нь канон бүлгүүдэд:
> greenfield → `specs/extended/workflow/10-adoption-guide/01-adopting-zeespec-from-scratch.md`,
> brownfield → `specs/extended/workflow/10-adoption-guide/02-onboarding-existing-project.md`.
> Refactor хувилбарт канон бүлэг одоогоор **байхгүй** — N=2 хаагдсаны дараа нэмэх кандидат
> (content freeze, `ROADMAP-v4.md` standing rule 1).

**Нийтлэг томьёо:** орох цэг л өөр — идэй → *дизайны спек* / код → *археологийн спек* /
спек → *аюулгүйн тор*. Тогтворжсон хойноо гурвуулаа ижил хэмнэлд ордог:

> бичих үед автомат чиглүүлэлт + commit-ийн өмнө `/zeespec:check` + сард `/zeespec:drift`
> + шийдвэр бүрд `/zeespec:adr`

---

## 1️⃣ Шинэ төсөл — идэйн түвшнээс

> **Мөн чанар:** Идэйн бүсэд та жолоодно (ZeeSpec энэ давхаргыг зориудаар хамардаггүй —
> Zachman row 1), модуль үүссэн цагаас methodology жолоодно. Спек = дизайн; бүх зүйл
> 🚧 DESIGN-ээс төрж, код буухаар ✅ IMPL болж эргэнэ.

| # | Алхам | Хэрэгсэл |
|---|---|---|
| 1 | Идэй → 1 хуудас (асуудал · хэрэглэгч · амжилтын шалгуур · v1-д юу орохгүй) | `/ecc:plan-prd` эсвэл энгийн ярилцлага |
| 2 | Зохицуулалттай домэйн бол: "бодит хэрэглэгчтэй болоход ямар дүрэм үйлчлэх вэ?" | R4 агент (эрт дуудаж болно) |
| 3 | Модулийн задаргаа — 3-7 модуль, аль нь инвариант-баялаг вэ | Ярилцлага (MODULES.md загвар) |
| 4 | Цөм модулиос эхэл | `/zeespec:init <модуль>` |
| 5 | `why.md` + `what.md` бич — бүх INV 🚧 DESIGN, мэдэхгүй бүхэн `gaps.md`-д OPEN | `/zeespec:author` |
| 6 | Кодоо бичүүл — Claude спекээс урган бичнэ, INV иш татна, gap дээр зогсоно | (автомат) |
| 7 | Feature дуусмагц: таг эргүүл (🚧→✅ + file:line + test pointer) | "тагуудыг кодтой тулгаж шинэчил" |
| 8 | Commit-ийн өмнө | `/zeespec:check` |
| 9 | Эхний долоо хоногтоо CI gate (warn mode) | `scripts/ci-drift-gate.sh` + workflow |

**Анхаар:**
- Спек кодоос **1-2 долоо хоногоос илүү бүү түрүүлэг** — "эхлээд бүгдийг спеклэе" бол №1 үхлийн урхи.
- Material шийдвэр бүрд `/zeespec:adr` — 2 минут; хойшлуулбал ШАЛТГААН нь мартагдана.

## 2️⃣ Хуучин код — ажиллаж байгаа систем

> **Мөн чанар:** Урсгал эсрэгээрээ — код canonical, спек нь археологи. Бүх claim эхнээсээ
> `file:line`-тэй, статусууд шалгалтаар батлагдана. Эхний өгөөж нь бодит bug олдох
> (пилот 1: 4 bug, NewsApp: 1 P0). Энэ бол арга зүйн төрөлх нутаг.

| # | Алхам | Хэрэгсэл |
|---|---|---|
| 1 | Байгаа доc-уудын audit: шинэчлэгдсэн үү · бүрэн үү · итгэдэг үү? Үгүй бол "NOT authoritative" гэж тунхагла | 1 цагийн ажил |
| 2 | Код-үндэслэлтэй модулийн зураглал — import graph-аас, доc-оос БИШ | `/zeespec:init` + MODULES.md |
| 3 | **Хамгийн эрсдэлтэй** модулиос эхэл (incident их · мөнгө/PII · "зөвхөн би мэднэ") | — |
| 4 | Lite-аар extraction: entity ← ORM/schema, алгоритм ← service, ойлгомжгүй код ← `gaps.md`-д асуулт | `/zeespec:author` |
| 5 | Хамгийн чухал нэгийг бүрэн pipeline-аар — олдвор энд гарна | `/zeespec:promote <модуль>` |
| 6 | Олон жилийн шийдвэрүүдийн учрыг сэргээ (эхний модульд 5-10 ш) | `/zeespec:adr` (retroactive) |
| 7 | Gate CI-д суулга — эхний өдрөөсөө бодит утгатай | ci-drift-gate (warn) |
| 8 | Хэмнэл: нөлөөлөх PR бүрд спек шинэчлэлт + check; сард drift | `/zeespec:check` · `/zeespec:drift` |

**Анхаар:**
- **Сард 1 модулиас бүү хэтэр.** Спек кодоо гүйцэхгүй, арга зүй чулуу болж зүүгдэнэ.
- Спек бичилт шинэ feature-ийг хэзээ ч блоклох ёсгүй — зэрэгцээ яв, HEAD-OF-MAIN дээр бич,
  2 долоо хоногоор time-box хий.
- Хуучин wiki/доc-оо бүү хая — эх сурвалж болгож иш тат (migrate, don't replace).

## 3️⃣ Refactor хийж байгаа үед

> **Мөн чанар:** Спек бол refactor-ийн **аюулгүйн тор** — "юу өөрчлөгдөж БОЛОХГҮЙ вэ" гэдгийг
> инвариантууд барина. Ишлэлүүд эвдрэх нь хэвийн (код нүүдэг — Type 1 drift), инвариант
> эвдрэх нь bug.

| # | Алхам | Хэрэгсэл |
|---|---|---|
| 1 | Спек байгаа юу? Үгүй бол **эхлээд** Lite-аар dock — одоогийн зан төлөвийг хадгал (characterization) | `/zeespec:init` + author |
| 2 | Цэвэр сууриас эхэл — одоогийн байдлаа баталгаажуул | `/zeespec:check <модуль>` |
| 3 | P0/P1 инвариантуудын **Test pointer**-уудыг шалга — тестгүй дүрэм бол refactor-ийн ӨМНӨ тест бич (v4.0 test-pointer дүрмийн яг зориулалт) | `what.md` `Test` багана |
| 4 | `gravity.md`-ийн HW-уудыг унш — хөндлөн дүрмүүд refactor-д хамгийн амархан эвдэрдэг | — |
| 5 | Ангил: **механик** (нэр/байршил/бүтэц; зан төлөв хэвээр) үү, **дизайн** (хил/state machine өөрчлөгдөнө) үү? Дизайн бол эхлээд ADR | `/zeespec:adr` |
| 6 | Refactor-оо хий — Claude INV-уудыг иш татаж, HW-г сахиж явна | (автомат) |
| 7 | Ишлэлүүд эвдэрсэн нь хүлээгдэж байсан Type 1 — "спекийн file:line ишлэлүүдийг шинэ байршилд тулгаж шинэчил" | gate + Claude |
| 8 | Тестүүд ногоон + check давсан = инвариантууд амьд, статусууд ✅ хэвээр | `/zeespec:check` |
| 9 | Том refactor байсан бол семантик баталгаажуулалт | `/zeespec:drift` эсвэл `/zeespec:review` |

**Анхаар:**
- **"Refactor хийнгээ жоохон зан төлөв сайжруулчихъя" бол хамгийн аюултай хольц** — спекийн
  нүдээр Type 1 биш Type 3 өөрчлөлт болж, ADR + спек шинэчлэлт шаардана. Хоёр төрлийг нэг
  PR-д бүү холь.

## Командын лавлах (бүх хувилбарт нийтлэг)

| Мөч | Команд | Давтамж |
|---|---|---|
| Шинэ төсөл / шинэ модуль | `/zeespec:init [module]` | нэг удаа |
| Спек бичих / өргөтгөх | `/zeespec:author` | модуль бүрд |
| **Commit хийхийн өмнө** | `/zeespec:check <module>` | **байнга ★** |
| "Юу намайг блоклоод байна?" | `/zeespec:gaps` | долоо хоногт |
| Спек бүрэн шалгуулах | `/zeespec:review <module>` | том өөрчлөлтийн дараа |
| Чухал модулийг Full болгох | `/zeespec:promote <module>` | 1-2 модульд л |
| "Код спекээсээ зөрчихсөн үү?" | `/zeespec:drift <module>` | сард / сэжиг төрөхөд |
| Шийдвэр гаргасны дараа | `/zeespec:adr` | шийдвэр бүрд |
| Methodology хувилбар гаргасны дараа | `/zeespec:metrics` | хувилбар бүрд |
